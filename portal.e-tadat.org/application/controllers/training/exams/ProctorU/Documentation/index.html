Quick Start Guide

Introduction

The aim of this document is to give a quick explanation of the functionality of, and how to use, ProctorU's PHP based API wrapper. Further documentation which touches on things such as properties, method arguments, return values, etc., can be found within the source code. You can identify this information by looking for docblock comments:

					/** 

					 * This is an example of a docblock comment 

					 * 

					 * They span multiple lines and tell you really cool things about properties and methods. 

					 */

For questions and problems that are not addressed within the documentation that is shipped with this library, our tech team can be reached by e-mail at techteam@proctoru.com. That being said, let's begin.

Installation

The bulk of this library is ready to use out of the box, as its classes are designed to function (somewhat) independently. However there are two lines that you are required to edit before it will work with ProctorU's API.

First, navigate to root/ProctorU/config.php. You'll notice that at the top of the script there are two lines:

					define('API_URL', 'https://apitest.proctoru.com/'); 

					define('API_KEY', null);

API_URL stores the base URL which requests will be submitted to. It's important to note that the trailing slash is required. Its value defaults to ProctorU's testing API URL, where any requests that you don't intend to be pushed into production will go. This does not actually need to be changed until your application enters production.

API_KEY stores, well, your API key. This value will be sent in the HTTP request header as Authorization-token to authenticate your application. Its default value is null.

Creating a New Instance

Instantiating the wrapper in your script is a simple process. First, include the configuration file:

						require_once 'ProctorU/config.php';

Then create the Wrapper object, supplying the two configuration constants that were discussed above as arguments:

						$api = new ProctorU\Wrapper(API_URL, API_KEY);

Making Calls

Any working endpoint of the API is automatically a method of the Wrapper class. This is achieved through the magic method __call(). A call to the API will either return false, in the event of an error, or an object of the Response class.

						$response = $api->getEmailExist(array('email' => 'student123@institution.com'));

Passing Arguments and Headers

Wrapper::__call() accepts two arrays as arguments: Controller arguments and HTTP headers. The first array must be associative and the other must simply be an index. For example, here's the same call as previous, with a custom User-agent header:

						$params = array('email' => 'student123@institution.com'); 

						$headers = array('User-agent: ProctorU Awesome Example'); 

						


						$response = $api->getEmailExist($params, $headers);

The time_sent parameter and Authorization-token header will be automatically injected into the request, so you do not need to input them.

Gathering Data

Thanks to the DataMethods trait, which is used by the Request and Response classes, you can easily gather data pertaining to both the HTTP request and response. If you wish to see everything about a particular object, echo it onto the page:

						echo $response; 


						// Output: array ( 'data' => array ( 'schoolname' => 'Internet Testing Systems - ETS', ), 'message' => '', 'response_code' => 1, 'time_sent' => '2014-04-01T18:37:01Z', )

When accessing single elements, you can call the key as if it were a property of the class. For example, let's take the school name that was returned by the previous call:

						echo $response->data['schoolname']; 


						// Output: Internet Testing Systems - ETS

Catching Exceptions

Coupled with this library is a custom child of PHP's Exception class. This class was written to provide a means of logging errors. Here's an example of how it works:

						$params = array('email' => 'thisuserdoesntexist@institution.com'); 
						


						try { 

						    $response = $api->getEmailExist($params); 

						} catch(ProctorU\Exception $e) {

						    $e->log(); 

						}

Note that it's only important to prefix the name of the class (Exception) with ProctorU\ if your script is executing outside of the ProctorU directory. This behavior is due to PHP's namespacing feature.

Logs are stored within root/ProctorU/Logs on a daily basis -- meaning, if you're testing your application at 11:59p.m. on December 31st, and catch an Exception, it will be stored in 12-31.log. Then, one minute into the future, you catch another exception... this time you'll find it in 01-01.log. Make sense? Good.
